8a. **Bootstrap INSERT policies for first-family creation** — the generic `*_insert` policies use `with check (family_id = private.current_family_id())` which evaluates to FALSE during the Family Creation Wizard's first INSERTs (because `private.current_family_id()` returns NULL until the user has a row in `public.members`). Add these two bootstrap policies AFTER the generic block (they sit alongside the generic INSERT policies; Postgres applies the union of all permissive policies):

   - `create policy members_insert_bootstrap on public.members for insert to authenticated with check (auth.uid() is not null and not exists (select 1 from public.members m where m.auth_user_id = auth.uid()));` — allows an authenticated user to INSERT their FIRST member row when they have no member rows yet. Once they have one, this policy stops applying and the generic `members_insert` policy (gated on `family_id = private.current_family_id()`) takes over.

   - `create policy family_settings_insert_bootstrap on public.family_settings for insert to authenticated with check (family_id in (select id from public.families where created_by = auth.uid()));` — allows INSERT into `family_settings` when the authenticated user is the `created_by` of that family (i.e., they just inserted the family row in the same wizard submit). The window is narrow: only the creator of a family can bootstrap its settings, and only before any other state exists.

   These two bootstrap policies close the chicken-and-egg gap in the Family Creation Wizard (Plan 05 Task 5.2) without requiring service-role bypass. They are intentionally narrow: `members_insert_bootstrap` ONLY applies when the user has zero member rows (so it cannot be exploited to inject into someone else's family), and `family_settings_insert_bootstrap` ONLY applies when the user is the family's `created_by`.
9. `allowed_emails` policies: SELECT-own (`lower(email) = lower((select auth.jwt() ->> 'email'))`), INSERT-parent (`private.auth_is_parent()`), DELETE-parent. No UPDATE policy (allowlist rows are immutable; revoke = delete).
10. `family_links` SELECT policy: union over both sides (`family_a_id = private.current_family_id() OR family_b_id = private.current_family_id()`). No INSERT/UPDATE/DELETE in Phase 1 — Phase 4 adds those when the linked-family UI exists.

Do NOT include `handle_new_user()` trigger here — that ships with MEMB-07 in Phase 2. Do NOT include `chore_streaks` views, `notifications_queue` triggers, or `pg_cron` schedules — those ship in Phases 3 and 6.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (anon JWT) → Postgres | Every read/write transits RLS; cross-tenant access must be blocked at the database level. |
| migration author → production DB | A bad migration can drop production data; mitigations are git review + the human checkpoint Task 2.2. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Cross-tenant SELECT via spoofed `family_id` | mitigate | RLS uses `private.current_family_id()` derived from `auth.uid()` join in the `members` table — cannot be spoofed by the client. Verified by `tests/integration/rls-isolation.test.ts` (RED until Plan 02 push completes; GREEN after). |
| T-02-02 | Elevation of Privilege | RLS helper recursion / search_path attack | mitigate | `private.current_family_id()` and `private.auth_is_parent()` are `SECURITY DEFINER` in the `private` schema with `set search_path = ''`. The `private` schema is not exposed via PostgREST. |
| T-02-03 | Information Disclosure | Allowlist enumeration via SELECT on `allowed_emails` | mitigate | RLS SELECT-own policy restricts to `lower(email) = lower((select auth.jwt() ->> 'email'))` — user sees only their own row, never the whole list. |
| T-02-04 | Tampering | Unauthorized allowlist mutation | mitigate | INSERT and DELETE policies on `allowed_emails` require `private.auth_is_parent() = true`. No UPDATE policy (allowlist rows are immutable). |
| T-02-05 | Information Disclosure | Realtime channel exposure of other families' rows | mitigate | Realtime postgres_changes events respect RLS — an attacker who subscribes to `family:xyz` channel still only receives events for rows where their RLS check passes. Server-side enforcement. |
| T-02-06 | Tampering | Bootstrap allowlist contains wrong email | mitigate | Human checkpoint Task 2.2 forces a manual review of the 5 bootstrap emails before push. |
| T-02-07 | Denial of Service | Unlimited family creation by a single auth user | accept | `families_insert` policy uses `with check (true)` per RESEARCH.md Open Question 4 (RESOLVED). Accepted because (a) the family is one trusted unit, (b) Phase 2 + Stripe billing will gate this, (c) RLS still prevents cross-family reads. |
| T-02-08 | Elevation of Privilege | `members_insert_bootstrap` policy lets a user insert their first member row anywhere | mitigate | The `not exists` predicate ensures the policy is consumable exactly once per `auth.uid()`. Once any member row exists for that user, the bootstrap stops applying and the generic `family_id = private.current_family_id()` gate takes over. The wizard always pairs the member INSERT with a freshly-created family the same user just created, so the inserted `family_id` is provably the user's own. |
| T-02-09 | Elevation of Privilege | `family_settings_insert_bootstrap` policy lets a user inject settings into a family they did not create | mitigate | The `with check` predicate requires `family_id in (select id from public.families where created_by = auth.uid())`. Only the family's `created_by` (set at family INSERT time to `auth.uid()`) can satisfy this. The window closes after the wizard completes because future `family_settings` writes go through the generic `*_update` policy. |
| T-02-SC | Tampering | npm/supabase CLI supply-chain | mitigate | Supabase CLI installed from Homebrew (RESEARCH.md §Environment Availability). No new npm packages added in this plan. |
</threat_model>
