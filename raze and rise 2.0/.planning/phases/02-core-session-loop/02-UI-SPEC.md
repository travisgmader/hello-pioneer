# Phase 2 — UI Design Contract

> Visual and interaction contract for the Core Session Loop (active workout: start, log, rest, complete with Anubis).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inherits and extends Phase 1.** All Phase 1 tokens (colors, spacing, typography, motion, radius) are unchanged. Phase 2 introduces NEW tokens (numeric/mono role, set-state semantic colors, timer pill surface) and NEW components (SessionScreen, ExerciseCard, SetRow, RestTimerPill, AnubisOverlay, BodyMap, ExerciseSwapModal, SessionNoteSheet). The Phase 1 "Phase-to-Phase Token Stability Contract" is honored — no Phase 1 token is redefined.
>
> Aesthetic target (DESIGN-02): **Whoop / Strong**. Dark. Data-dense. Serious-athlete. Compact set rows, monospaced numerics, restrained accent. No card-with-heavy-padding lifestyle UI.

## Spacing Scale

Phase 2 uses the Phase 1 8-point scale unchanged. No new spacing tokens introduced.

| Token | Value | Phase 2 Usage |
|-------|-------|---------------|
| `xs` | 4px | Set-row internal vertical gap, dot spacing inside `✓✓✓✗` history glyph, gap between weight value and `lbs` unit label |
| `sm` | 8px | Set-row internal horizontal padding, gap between set rows inside a card, gap between Go and No-go buttons, RestTimerPill internal gap |
| `md` | 16px | ExerciseCard internal padding, screen edge horizontal padding (carry), gap between ExerciseCards in the FlashList, RestTimerPill height baseline |
| `lg` | 24px | Section padding inside expanded-set form, gap above the first ExerciseCard from header |
| `xl` | 32px | Top spacing from session header to first card on first scroll position |
| `2xl` | 48px | (unused in Phase 2 active session — reserved for empty-state spacing if a template is missing) |
| `3xl` | 64px | (unused in Phase 2) |

**Exceptions (Phase 2 — declared):**

- **Set row height: 56pt minimum** (taller than 44pt because the row contains weight input + go/no-go + chevron + history glyph in one horizontal scan line — needs density per Whoop/Strong aesthetic without sacrificing tap targets)
- **Weight TextInput inside SetRow: 48pt height, 88pt min-width** (must accept 3–4 digit weight + decimal; min-width prevents layout shift when value changes from 95 to 225)
- **Go / No-go buttons inside SetRow: 44pt height each, 56pt min-width** (square-ish; full text "Go" / "No-go" inside)
- **Chevron expand button per row: 44×44pt** (hit area via `hitSlop`, visible icon 20px Lucide `ChevronDown` / `ChevronUp`)
- **RestTimerPill height: 64pt** (above bottom safe area, contains 1 line of timer numerics at 28px + control buttons at 36pt) — single declared exception for a floating overlay
- **RestTimerPill horizontal margin: 16pt** (matches screen edge padding)
- **RestTimerPill bottom offset: 16pt** above the bottom safe area inset
- **Complete Workout button (in session header): 36pt height, compact** — the only place in the app where the primary CTA is below the 48pt minimum, justified because (a) it's not the most common action on screen and (b) header density is intentional per DESIGN-02
- **Body Map (injury flag) tap targets: 32×32pt visible muscle group regions, expanded to 44×44pt hit area via `hitSlop`** — anatomical SVG paths cannot be 44pt visually without breaking the front/back-view layout

All exceptions are declared. None are arbitrary; each maps to either iOS HIG (44pt) or Whoop/Strong density precedent.

## Typography

**Phase 2 active type scale: 4 sizes (12 / 16 / 24 / 28).** Display (32px) is a Phase 1 carry token declared in `tailwind.config.js` but NOT used in Phase 2 screens — excluded from the active count. One new role added (`numeric-large`).

| Role | Size | Weight | Line Height | Font | Phase 2 Usage |
|------|------|--------|-------------|------|---------------|
| Caption (carry) | 12px | 400 | 1.4 (17px) | Manrope | Previous performance text "185 lbs · ✓✓✓✗", set index "Set 1 / 3", RPE label "RPE", warm-up flag label, quick-tag chip labels, body map muscle name on tap, set count "5 sets" in card header |
| Body (carry) | 16px | 400 | 1.5 (24px) | Manrope | Session-level note text, exercise name in ExerciseCard header, expanded-set form labels (RPE, Warm-up, Note), free-text note input |
| Body emphasis (carry) + tabular-nums | 16px | 700 | 1.5 (24px) | Manrope + `fontVariant: ['tabular-nums']` | "Complete Workout" button label, exercise name when current/in-progress, selected quick-tag chip label. **Also used for all numeric values in set rows** (weight input display, RPE selected value, set count) — same 16px/700 size with `tabular-nums` applied via `NumericText` component. |
| Heading (carry) | 24px | 700 | 1.25 (30px) | Noto Serif | Session header day label ("Push", "Pull", "Legs", "AF PT Prep") — single brand moment per session |
| **Numeric large (NEW)** | 28px | 700 | 1.1 (31px) | Manrope + `fontVariant: ['tabular-nums']` | Rest timer countdown digits in RestTimerPill ONLY (`2:45`, `0:30`, `0:00`) — must read at arm's length across the gym. No other 28px usage in Phase 2. |
| Display (carry — Phase 2 unused) | 32px | 700 | 1.2 (38px) | Noto Serif | **Not used in Phase 2.** Declared in `tailwind.config.js` for Phase 1 parity; excluded from the 4-size active count. Anubis overlay does not use Display (Lottie renders the visual). |

**Tabular-nums policy:** Apply `fontVariant: ['tabular-nums']` via the `NumericText` component on ANY Text displaying weight, reps, time, RPE, or set count. Prevents digit-width jitter in stacked set rows. All numeric values use Body emphasis size (16px/700) except the timer pill countdown which uses Numeric large (28px/700). No 20px intermediate size — consolidating to 16px for dense set-row data keeps the scale tight and consistent with the Whoop/Strong data-dense aesthetic.

**No new font family in Phase 2.** JetBrains Mono was considered and rejected — `fontVariant: ['tabular-nums']` achieves column alignment without a 200KB font asset. Documented for the executor: do NOT add JetBrains Mono.

**`allowFontScaling={false}` policy continues** from Phase 1 for parity with the walking skeleton. Dynamic Type lands in Phase 6.

### Phase 2 Accent reserved list (EXTENDS Phase 1)

Phase 1's reserved list stays valid. Phase 2 ADDS the following:

1. **Currently-active ExerciseCard** — left-edge 2px accent bar + `accent-dim` background tint to indicate "this is the exercise the user is on right now" (defined as: the topmost exercise with at least one unfilled set, or the exercise that the user last interacted with — whichever is later)
2. **Go button selected state** — accent text "✓ Go" + accent-dim background
3. **Weight input focused state border** — `border-strong` (already on the Phase 1 reserved list — restated for Phase 2 set rows)
4. **RestTimerPill border (non-zero state)** — `border-strong` border to make the pill visually pop from the dark background without using accent fill (accent fill is reserved for the zero state)
5. **RestTimerPill background (zero state)** — full `accent` fill for the 3-second flash-and-dismiss
6. **Complete Workout button** — primary CTA (extension of Phase 1's "primary CTA = accent bg" rule)
7. **Body Map: selected sore muscle group** — `accent-dim` fill on the muscle region SVG path + `border-strong` outline

**Accent is NEVER used for (Phase 2 additions to Phase 1 ban list):**

- The chevron expand icon (use `fg-muted` — interactive but passive)
- The Skip button on the RestTimerPill (use `fg-muted` text, transparent background — secondary action)
- The ±30s buttons on the RestTimerPill (use `fg` text — interactive but neutral)
- The session-level note text area (use `fg` text, `bg-input` background — pure data entry)
- The set index number "Set 1 / 3" (use `fg-muted`)
- The previous performance text (use `fg-muted`)
- The set row weight value when NOT focused (use `fg`)

## Motion / Transitions

Carry from Phase 1. Phase 2 opts into ALL four declared timings plus adds ONE timer-specific timing.

| Token | Duration | Easing | Phase 2 Usage |
|-------|----------|--------|---------------|
| `instant` | 0ms | — | Set state change (null → go → no-go → null) — must feel like a physical button, not animate |
| `fast` | 150ms | ease-out | Go/No-go button press feedback, chevron rotation on expand, weight input focus border color shift, quick-tag chip selected color shift |
| `default` | 200ms | ease-out | ExerciseCard active-state border + tint transition when scrolling pulls a new exercise into "current" position, expanded-set form expand/collapse height |
| `slow` | 300ms | ease-in-out | RestTimerPill enter (slide up from bottom) and exit (fade + slide down) |
| **`timer-tick` (NEW)** | 1000ms | linear | RestTimerPill countdown bar progress fill — exactly 1 second per tick, linear easing (anything else feels wrong on a clock) |
| **`anubis-fade` (NEW)** | 600ms | ease-in-out | Anubis overlay fade-in over session screen on Complete Workout tap, fade-out from Anubis to Dashboard after Lottie completes |

**Anubis sequence (locked):**

1. User taps "Complete Workout"
2. Session screen captures completion intent, kicks off PowerSync writes in the background
3. Anubis overlay fades IN over 600ms (`anubis-fade`) on top of the session screen
4. Lottie animation plays once (no loop) — duration determined by the JSON asset (target: 2–3 seconds)
5. Anubis overlay fades OUT over 600ms (`anubis-fade`) as the Dashboard fades in beneath
6. Dashboard reflects the new rotation pointer (next day's workout)

Total user-perceived duration from tap → Dashboard: ~3.5 to 4 seconds. PowerSync writes complete during step 4 (the Lottie playback is the visual cover for the disk write).

**`useReducedMotion()` policy (carry from Phase 1):** When ON, the Anubis Lottie is replaced with a static frame held for 600ms before the fade — the Lottie still loads but does not play. RestTimerPill enter/exit becomes `instant`. Countdown bar still animates (`timer-tick` linear is essential for the clock affordance).

## Interaction Patterns

### Set Logging Flow (the central loop)

1. User opens the workout → BodyMap (optional, can skip) → SessionScreen
2. SessionScreen mounts, first ExerciseCard is marked as current (`accent` LeftEdgeBar + `accent-dim` tint)
3. User taps the WeightInput on Set 1 → keyboard opens → input is focused (`border-strong`) → user types weight OR taps PrevPerformanceLink to auto-fill
4. User taps `✓ Go` or `✗ No-go` on Set 1 → light haptic → button enters selected state (`set-go` or `set-nogo` colors) → set is committed to PowerSync → row gets LeftEdgeBar in the matching color
5. If set is marked "Go" (i.e., a working set, not warm-up): **RestTimerPill auto-opens** (slide up 300ms) with the rest duration from template (or global default if no override)
6. User can: (a) wait — countdown ticks every second; (b) tap ±30s to adjust; (c) tap Skip to dismiss immediately; (d) continue interacting with the list (pill is non-blocking)
7. At zero: pill morphs to zero state, vibrates, sounds, holds 3 seconds, fades out
8. If app is backgrounded during countdown: a local notification fires at zero with copy "Rest complete — back to it." (no action button required)
9. After rest, next SetRow on the same exercise becomes the natural target (or, in supersets, scroll auto-jumps to the paired exercise per D-11/WORKOUT-11)
10. User repeats through all sets across all exercises
11. User taps "Complete Workout" in header → AnubisOverlay fades in → Lottie plays → fades to Dashboard

### Keyboard Behavior

- Weight inputs use `keyboardType="decimal-pad"` (numeric keypad with decimal, no return key needed — value commits on blur)
- Session note + set free-text use default keyboard with `returnKeyType="default"`
- Tapping the SessionScreen background dismisses the keyboard
- KeyboardAvoidingView wraps the SessionScreen so the focused SetRow stays visible above the keyboard

### Background / Foreground Handling

- On AppState change to `background`: do nothing visible — PowerSync continues writes, RestTimerPill schedules a local notification if a timer is running and the zero time has not yet passed
- On AppState change to `active`: RestTimerPill recomputes remaining time from a stored `startedAt` timestamp (not from a ticking counter) — guarantees accuracy across backgrounding
- If the user closes the app entirely during a session: SessionScreen re-mounts with the session intact (per WORKOUT-18, PowerSync local persistence). No data loss, no recovery prompt.

### Loading States

- **No skeleton screens on SessionScreen.** PowerSync read is local (<10ms typical); the FlashList renders the data without a perceptible gap. If data takes >300ms to load, a Phase 1 `Spinner` shows centered with no surrounding copy.
- **ExerciseSwapModal:** Search has no loading state — search is over the local exercise library (PowerSync local table)
- **Complete Workout:** No loading state on the button itself. The AnubisOverlay IS the loading state — Lottie playback is the visual cover for PowerSync writes.

### Error State Patterns

| Scope | Pattern |
|-------|---------|
| Weight input — invalid (negative, >999.9, non-numeric) | Inline HelperText (error) under WeightInput, value reverts to previous on blur |
| Timer audio/notification permission denied | Silent fallback — visual countdown still works, haptic still fires, no error banner shown (timer is functional without sound) |
| PowerSync write failure | Silent retry — PowerSync queues writes; no user-visible error during a session. If queue depth exceeds threshold, a small caption appears under SessionHeader: `Sync paused — your sets are saved locally.` in `fg-muted`. |
| Anubis Lottie fails to load | Fallback: 600ms blank `bg`-colored overlay, then proceed to Dashboard (session is already committed). No error UI. |
| Exercise swap — exercise library failed to load | Inline HelperText inside the modal: `Can't load exercises. Try again.` with a "Retry" ghost Button. |

**Phase 1 inline-only rule continues in Phase 2.** No toasts, no snackbars, no `Alert.alert` during a session. The single Phase 1 exception (Sign Out from Settings) does not apply during a workout.
