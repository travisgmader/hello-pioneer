# Phase 1 — UI Design Contract

> Visual and interaction contract for the Foundation phase (Auth, Onboarding, 5-Tab Nav, Dashboard stub).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This is the **first phase** of v2 — these tokens establish the design system that Phases 2-6 must extend without breaking. The brand DNA is carried from v1 (gold-on-obsidian, Manrope + Noto Serif, gothic "Raze and Rise" energy) but rebuilt for native iOS/Android with NativeWind v4.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (NativeWind v4 + custom token system — shadcn is web-only and not applicable to React Native) |
| Preset | not applicable |
| Component library | none (RN primitives wrapped in NativeWind classes; custom components in `src/components/`) |
| Icon library | `lucide-react-native` (tree-shakable, stroke-based, matches the precise gothic line aesthetic — avoids the bloated 60s-corporate look of `react-native-vector-icons/MaterialIcons`) |
| Font (sans) | Manrope (carry from v1) — loaded via `expo-font` at app startup; weights 400, 700 |
| Font (serif/display) | Noto Serif (carry from v1) — used ONLY for h1/h2 brand moments; weight 700 |
| System font fallback | iOS SF Pro / Android Roboto via `system-ui` |
| Theme default | Dark (DESIGN-01 — light mode toggle deferred to Settings, dark is shipping default) |
| New Architecture | enabled (SDK 55 mandatory) |
| Animation library | `react-native-reanimated` v4 (required for NativeWind v4 transitions per DESIGN-04) |

**Token export pattern (planner directive):** All values below MUST be declared in `tailwind.config.js` under `theme.extend.colors`, `theme.extend.fontSize`, `theme.extend.spacing` so they are reachable from NativeWind classes (`bg-bg`, `text-fg`, `p-md`, etc.). No raw hex values in component code — components reference tokens only.

## Spacing Scale

Declared values (multiples of 4, 8-point grid):

| Token | Value | Usage |
|-------|-------|-------|
| `xs` | 4px | Icon-to-label gap, inline badge padding |
| `sm` | 8px | Compact element spacing, button internal vertical padding, gap between stacked social buttons |
| `md` | 16px | Default element gap, screen edge horizontal padding, form field vertical gap |
| `lg` | 24px | Section padding, card internal padding, gap between auth fields and divider |
| `xl` | 32px | Layout gaps, gap between onboarding progress bar and step content |
| `2xl` | 48px | Major section breaks, top of step content from progress bar |
| `3xl` | 64px | Empty state vertical padding, hero brand spacing on auth screen |

**Exceptions (DECLARED — all icon-only touch targets):**
- Tab bar icon hit area: 44pt minimum (iOS HIG) / 48dp minimum (Android Material) — implemented via `hitSlop` not padding
- Auth screen toggle tabs: 44pt touch height per tab
- Social login button (Google / Apple): 48pt vertical height (locked by Apple HIG for Apple Sign-In button)
- Onboarding back chevron: 44×44pt with 8px visible icon centered

Screen edge horizontal padding is `md` (16px) on phones. Vertical safe area (notch / home indicator) handled by `react-native-safe-area-context` — never hardcoded.

## Typography

Two weights only (400 regular, 700 bold). Four sizes — no inline overrides permitted.

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Caption | 12px | 400 (regular) | 1.4 (17px) | Manrope | Helper text under inputs, progress "Step 2 of 4", tab labels, forgot password link, inline text links |
| Body | 16px | 400 (regular) | 1.5 (24px) | Manrope | Input text, paragraph copy, button labels (secondary buttons), list items, field labels above inputs |
| Body emphasis | 16px | 700 (bold) | 1.5 (24px) | Manrope | Selected state labels, primary CTA button text, emphasized inline copy |
| Heading | 24px | 700 (bold) | 1.25 (30px) | Noto Serif | Onboarding step titles ("Set up your profile"), Settings section headers |
| Display | 32px | 700 (bold) | 1.2 (38px) | Noto Serif | Auth screen "Raze and Rise" wordmark only — reserved brand moment |

**Heading font policy (Noto Serif):** Serif is reserved for h1/h2 brand moments only — onboarding step titles, auth wordmark, completion screens. All body, labels, button text, and dense data UI (coming in Phase 2) use Manrope. Mixing serifs into dense layouts breaks the workout-screen "readable at arm's length under gym lights" goal.

**No font scaling for accessibility in Phase 1:** Defer dynamic type / `allowFontScaling` policy to Phase 6 (Polish). Phase 1 ships with `allowFontScaling={false}` on all `Text` components to prevent layout breakage during walking-skeleton verification. (Logged as a known gap — a11y auditor will flag this in Phase 6.)

## Color

Dark-first palette. v1 brand DNA preserved: obsidian background + gold accent. Tightened for native (higher contrast on OLED, simpler ramps for NativeWind).

| Role | Token | Hex | Usage |
|------|-------|-----|-------|
| Dominant surface (60%) | `bg` | `#0A0A0B` | App background, onboarding step background, tab bar background |
| Secondary surface (30%) | `bg-elevated` | `#141416` | Input fields, cards, social button backgrounds, modal surfaces |
| Tertiary surface | `bg-input` | `#1C1B1B` | Pressed states of `bg-elevated` surfaces, divider segments behind "or" text |
| Border (default) | `border` | `rgba(212, 175, 55, 0.22)` | Input borders (unfocused), card borders, divider lines |
| Border (focus / strong) | `border-strong` | `rgba(212, 175, 55, 0.45)` | Focused input border, selected onboarding option card |
| Foreground primary | `fg` | `#E5E2E1` | Body text, input text, headings (when not gold) |
| Foreground muted | `fg-muted` | `#99907C` | Captions, helper text, inactive tab labels, placeholder text |
| Foreground subtle | `fg-subtle` | `#5C564B` | Disabled text, very low-emphasis labels |
| **Accent (10%)** | `accent` | `#F2CA50` | **See reserved list below** |
| Accent deep | `accent-deep` | `#D4AF37` | Hover/pressed state of accent elements; gold-frame border corners |
| Accent dim | `accent-dim` | `rgba(212, 175, 55, 0.10)` | Selected onboarding option background tint, focused input background tint |
| Destructive | `danger` | `#EF4444` | Auth error message text, destructive action labels (e.g., "Delete account") |
| Destructive dim | `danger-dim` | `rgba(239, 68, 68, 0.12)` | Inline error message background under input fields |
| Success | `success` | `#10B981` | "Account created" confirmation, migration complete state |
| Success dim | `success-dim` | `rgba(16, 185, 129, 0.12)` | Success message background |

### Accent reserved for (EXPLICIT LIST — never used outside this list):

1. Primary CTA button background (`Continue`, `Next`, `Get started`, `Sign in`, `Create account`)
2. Active tab indicator (color of the active tab's icon and label in the 5-tab bottom navigator)
3. Onboarding progress bar fill (the filled portion of the step-N-of-4 indicator)
4. Focused input border + caret color
5. Selected option border on Split Selector cards, Goal Selector cards, Unit Toggle (lbs/kg)
6. "Raze and Rise" wordmark text on the auth screen (gold-shimmer gradient using `accent-deep → accent → accent-deep`)
7. Forgot password link, "Sign up" / "Sign in" toggle link copy
8. Apple Sign-In button text color override is NOT permitted — Apple's HIG requires their button to render in its native white-on-black or black-on-white form. Apple button skips accent.

**Accent is NEVER used for:**
- Body text (use `fg`)
- Card backgrounds (use `bg-elevated`)
- Borders on unfocused elements (use `border`)
- Tab bar background (use `bg`)
- Icons in passive UI states (use `fg-muted`)

### Contrast verification (WCAG AA on dark background)

All combinations verified to meet AA (4.5:1 for normal text, 3:1 for large text):

| Foreground | Background | Ratio | Verdict |
|------------|------------|-------|---------|
| `fg` `#E5E2E1` | `bg` `#0A0A0B` | 14.8:1 | AA + AAA |
| `fg-muted` `#99907C` | `bg` `#0A0A0B` | 6.4:1 | AA |
| `fg-subtle` `#5C564B` | `bg` `#0A0A0B` | 2.9:1 | FAILS — use only for non-text decorative purposes (disabled affordance, never load-bearing copy) |
| `accent` `#F2CA50` | `bg` `#0A0A0B` | 11.8:1 | AA + AAA |
| `bg` `#0A0A0B` | `accent` `#F2CA50` | 11.8:1 | AA + AAA — primary CTA copy is `bg` on `accent` background |
| `danger` `#EF4444` | `bg` `#0A0A0B` | 5.1:1 | AA |
| `fg` `#E5E2E1` | `bg-elevated` `#141416` | 12.6:1 | AA + AAA |

**Constraint for executor:** `fg-subtle` is for borders/decorative shapes only, never for text the user needs to read. Disabled button labels use `fg-muted` at 60% opacity (`opacity-60`), not `fg-subtle`.

## Border Radius

| Token | Value | Usage |
|-------|-------|-------|
| `none` | 0px | Dividers, full-bleed sections |
| `sm` | 2px | Inputs, tight chips |
| `md` | 4px | Buttons, cards, social buttons (matches v1 sharp/gothic angle — no soft pill shapes) |
| `lg` | 8px | Modal sheets, large onboarding option cards |
| `full` | 9999px | Avatar/profile image circles only |

**No pill-shaped buttons.** The v1 brand uses sharp 4px corners deliberately — soft rounded UI reads as "lifestyle app" (MyFitnessPal, Fitbit). Raze and Rise is gothic / serious-athlete. Keep angles tight.

---

## Shadow / Elevation

Dark UIs do not get drop shadows in the conventional sense — shadows on `#0A0A0B` are invisible. Use **gold-tinted glow** for elevation cues and **inset/border treatment** for cards.

| Token | Value | Usage |
|-------|-------|-------|
| `shadow-none` | none | Default state — cards rely on `bg-elevated` color contrast |
| `shadow-glow` | `0 0 20px rgba(212, 175, 55, 0.15)` | Hover/pressed state of accent CTAs only |
| `shadow-glow-strong` | `0 0 24px rgba(212, 175, 55, 0.25)` | Focused state of primary CTA button |
| `shadow-modal` | `0 8px 32px rgba(0, 0, 0, 0.7)` | Bottom sheet modals (future Phase 2+) — not used in Phase 1 |

**Phase 1 uses `shadow-none` everywhere.** Glow tokens are declared so Phase 2 (workout session, Anubis completion overlay) can opt in without re-establishing the token system.

## Motion / Transitions (DESIGN-04)

| Token | Duration | Easing | Usage |
|-------|----------|--------|-------|
| `instant` | 0ms | — | State changes that must not animate (form validation errors appearing) |
| `fast` | 150ms | ease-out | Button press feedback, tab switch color change, input focus border color shift |
| `default` | 200ms | ease-out | Onboarding progress bar fill, selected card border |
| `slow` | 300ms | ease-in-out | Screen transitions (onboarding step → step via Expo Router stack `slide_from_right`) |

**150ms minimum on interactive elements** per DESIGN-04. All button presses, toggle changes, focus rings use `fast` or `default`. The rest timer animation comes in Phase 2.

**Reduce Motion respect:** Use `useReducedMotion()` from `react-native-reanimated` to disable screen-slide animations when the OS setting is on. Required for App Store submission (a11y compliance check happens in Phase 6 but the hook must be wired from Phase 1 to avoid retrofitting).

### Migration Progress Screen (`app/migration.tsx`)

Full-screen, NO tab bar visible (gated by root layout). Three render states:

**`in_progress`:**
- Heading "Bringing your history forward" — Noto Serif 24px
- Body "We're moving your v1 workouts into the new app. Hang tight." — Manrope 16px fg-muted
- Centered `Spinner` (accent color, 32px)
- No buttons — user cannot escape until success or failure

**`complete`:** (shown for ~1s before auto-routing to onboarding or tabs)
- Lucide `CheckCircle2` 32px in `success` color
- Heading "All set"
- Body "Your history is ready."

**`failed`:**
- Lucide `AlertTriangle` 32px in `danger` color
- Heading "Something went wrong"
- Body "We couldn't import your data. You can try again — your old workouts are safe."
- Primary Button "Retry" (full-width)
- Secondary Button "Contact support" (ghost — opens mailto link)

## Interaction Patterns

### Form Validation Timing

- **On blur** for individual field validation (email format, password length ≥8 chars, password match on Sign Up)
- **On submit** for cross-field validation and API-side errors (wrong password, email exists, network failure)
- Validation errors NEVER appear while the user is actively typing in a field — Wait for blur or submit
- Errors persist until either (a) the user re-focuses and edits the field, OR (b) the user successfully submits

**`zod` schema is the source of truth** (per RESEARCH.md stack) — same schema validates client and is mirrored in Edge Function `migrate-v1-user/index.ts` for migration triggers.

### Keyboard Behavior (iOS critical)

- All screens with text input wrap content in `KeyboardAvoidingView` (`behavior="padding"` on iOS, `"height"` on Android)
- Primary CTA buttons stay visible above the keyboard (sticky-bottom inside the safe area)
- Tapping outside an input dismisses the keyboard (`TouchableWithoutFeedback` wrapper around the screen)
- Return key configuration:
  - Email field → `returnKeyType="next"`, focuses Password field
  - Password field on Sign In → `returnKeyType="go"`, submits the form
  - Password field on Sign Up → `returnKeyType="next"`, focuses Confirm Password
  - Confirm Password → `returnKeyType="go"`, submits

### Haptic Feedback (Phase 1 baseline)

Use `expo-haptics` (already a stack dependency since SDK 55):

| Event | Haptic |
|-------|--------|
| Tab change | `Haptics.selectionAsync()` (light click) |
| Toggle change (Sign In ↔ Sign Up, lbs ↔ kg) | `Haptics.selectionAsync()` |
| Option card selected in onboarding | `Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Light)` |
| Primary CTA tap | NO haptic — adding haptic to every button press becomes noisy |
| Auth error | `Haptics.notificationAsync(Haptics.NotificationFeedbackType.Error)` |
| Auth success / onboarding complete | `Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success)` |

**Defer to Phase 2:** Workout set completion haptic (the meaningful one) — that's locked in by WORKOUT-03/06 design.

### Loading States

- **Buttons:** Loading variant replaces the button label with a Spinner in `bg` color (against the `accent` background). Button width does not change. Press is disabled during loading.
- **Migration screen:** Full-screen Spinner with explanatory copy (see Migration screen above) — never a skeleton screen, because there's no content layout to skeletonize.
- **Onboarding step transition:** No loading state — transitions are instant client-side state changes; data persists to MMKV synchronously.

**No skeleton screens in Phase 1** — there is no data-heavy content to skeletonize. (Skeletons land in Phase 2 for the active session's "Previous performance" column and Phase 3 for charts.)

### Error State Patterns

| Scope | Pattern |
|-------|---------|
| Field-level (invalid email, weak password) | Inline HelperText (error) under field |
| Form-level (wrong password, network error) | Inline HelperText (error) under primary CTA, above the divider |
| Screen-level (migration failed) | Full-screen error state with retry CTA |
| Critical (no network during auth) | Inline HelperText (error) under CTA + a "Retry" inline link in the body of the error message |

**Explicitly forbidden in Phase 1:** Toasts, snackbars, alert dialogs (`Alert.alert`), bottom-sheet error modals. CONTEXT.md Decision 3 locks inline errors only.

### Back Navigation Rules

| Screen | Back Behavior |
|--------|--------------|
| Auth | No back (gesture disabled) — this is the entry point |
| Onboarding step 1 (profile) | Back returns to Auth (sign out the user, clear MMKV onboarding state) |
| Onboarding step 2-4 | Back returns to previous step. Swipe-back gesture enabled (iOS). |
| Practice set (step 4) | Back returns to step 3. Skip button advances to Dashboard. |
| Migration screen | Back disabled (`gestureEnabled: false`) |
| Tabs (Dashboard/Workouts/Split/Progress/Settings) | Back is a no-op (tab bar is the navigation root) |

### Tap / Press Feedback

- All `Button` and `IconButton` components use `Pressable` with `opacity-80` on press
- Press duration < 100ms feels instant; press feedback uses NativeWind `active:opacity-80` utility
- Long-press is NOT used in Phase 1 (reserved for Phase 2 set row context menu)

## Accessibility

### Touch Targets

| Component | Minimum Size | Notes |
|-----------|--------------|-------|
| Buttons (primary, secondary, social) | 48pt height | Locked across all phases |
| Apple Sign-In button | 48pt height | Apple HIG requirement |
| Tab bar items | full column × 56pt | Well above 44pt minimum |
| Back chevron / IconButton | 44×44pt | Icon visually 24px, hit area expanded via `hitSlop` |
| Toggle / Chip / Option card | 44pt height minimum | Selected onboarding option cards 64pt to fit subtitle |
| Eye toggle in password field | 44×44pt | Hit area via `hitSlop`; visual icon 20px |
| Inline links (Forgot password, Back to sign in) | 44pt vertical hit area via `hitSlop` | Visual text remains 12px caption |

### VoiceOver / TalkBack Labels

All icon-only and ambiguous controls MUST have `accessibilityLabel` and `accessibilityRole`.

| Component | accessibilityRole | accessibilityLabel |
|-----------|-------------------|---------------------|
| Back chevron | `button` | `Back` |
| Eye toggle (show password) | `button` | `Show password` / `Hide password` (toggles) |
| Tab item | `tab` | Tab label (system-handled by Expo Router Tabs) |
| Toggle (Sign In/Up, lbs/kg) | `tab` | Option label + `accessibilityState={{ selected }}` |
| Goal Chip | `radio` | Goal label + `accessibilityState={{ selected }}` |
| Split option card | `radio` | Split name + subtitle joined with comma |
| Primary CTA Button (loading) | `button` | `accessibilityState={{ busy: true, disabled: true }}` |
| Inline error HelperText | `alert` | Full error message text |
| ProgressBar | `progressbar` | `accessibilityValue={{ min: 0, max: 4, now: currentStep }}` + `accessibilityLabel="Onboarding progress"` |
| Spinner | (no role — decorative) | parent screen provides context via heading |
| Migration screen | (heading role on h1) | Heading is the accessible name for the screen |

### Color Contrast

All foreground/background pairings verified to meet WCAG AA — see Color section table above. No text uses `fg-subtle` at any size.

### Dynamic Type

`allowFontScaling={false}` is the Phase 1 default on `Text` components to keep the walking skeleton stable. Logged as a known accessibility gap to be addressed in Phase 6 alongside App Store / Play Store submission.

### Reduce Motion

`useReducedMotion()` (from `react-native-reanimated`) wraps all screen-slide and progress-bar transitions. When ON: transitions snap to final state with no animation. Required wiring from day one — Phase 6 a11y check verifies it.

### Color-Independent State Communication

Selected states never communicate via color alone:
- Onboarding option cards: selected gets `border-strong` border (3px visual difference) PLUS `accent-dim` background tint PLUS a Lucide `Check` icon top-right
- Toggle pills: selected gets `accent-dim` background PLUS `border-strong` border PLUS `accent` text color PLUS weight 700 (vs. 400) PLUS `accessibilityState.selected = true`
- Active tab: `accent` icon/label color PLUS weight 700 (vs. 400) PLUS `accessibilityState.selected = true`

## Phase-to-Phase Token Stability Contract

Phases 2-6 MUST NOT redefine any token in this spec. They MAY:
- Add NEW tokens (e.g., Phase 2 will add `set-go` and `set-nogo` accent variants for set rows, `timer-fg` for the rest timer)
- Add NEW typography roles (e.g., Phase 2 will add `mono` role for weight/rep numeric displays — recommend JetBrains Mono or system monospace)
- Add NEW spacing tokens if absolutely needed (multiples of 4)
- Add NEW components — but every new component must reference Phase 1 tokens

Phases 2-6 MUST NOT:
- Change the meaning of any token defined here
- Use raw hex / px values outside the token system
- Introduce a second sans-serif or replace Manrope
- Introduce shadows other than the four declared in the Shadow section
- Override the inline-only error pattern (no toasts, no snackbars introduced later)
- Use accent color outside its explicit reserved list — if a new phase needs accent in a new place, that place must be added to this contract first
